# Metodologías ágiles y Scrum

Las metodologías ágiles priorizan la entrega incremental, la adaptación al cambio y la colaboración con el negocio. Scrum es un marco ágil muy usado: equipos trabajan en iteraciones cortas (sprints), con roles, artefactos y eventos definidos.

## Ágil vs enfoque tradicional

| Aspecto       | Tradicional (cascada)              | Ágil                                |
| ------------- | ---------------------------------- | ----------------------------------- |
| Planificación | Gran diseño al inicio              | Planificación continua y adaptativa |
| Entregas      | Al final del proyecto              | Incrementos frecuentes (semanas)    |
| Cambios       | Costosos una vez fijado el alcance | Cambios aceptados y priorizados     |
| Cliente       | Requisitos firmados al inicio      | Colaboración y feedback constante   |

## Valores del manifiesto ágil

- Individuos e interacciones sobre procesos y herramientas.
- Software funcionando sobre documentación exhaustiva.
- Colaboración con el cliente sobre negociación contractual.
- Respuesta al cambio sobre seguir un plan.

## Scrum: roles

| Rol                      | Responsabilidad principal                                           |
| ------------------------ | ------------------------------------------------------------------- |
| **Product Owner**        | Define el qué (backlog, prioridad); maximiza el valor del producto. |
| **Scrum Master**         | Facilita el proceso, elimina impedimentos, cuida que se siga Scrum. |
| **Equipo de desarrollo** | Diseña, construye y entrega el incremento dentro del sprint.        |

## Scrum: artefactos

| Artefacto           | Descripción                                                           |
| ------------------- | --------------------------------------------------------------------- |
| **Product Backlog** | Lista priorizada de todo lo que podría hacer el producto.             |
| **Sprint Backlog**  | Conjunto de ítems del Product Backlog elegidos para el sprint actual. |
| **Incremento**      | Resultado usable al final del sprint (cumple definición de hecho).    |

## Scrum: eventos

| Evento                   | Propósito breve                                                        |
| ------------------------ | ---------------------------------------------------------------------- |
| **Sprint**               | Iteración de tiempo fijo (1–4 semanas) donde se produce un incremento. |
| **Sprint Planning**      | Decidir qué se hará en el sprint y cómo.                               |
| **Daily Scrum**          | Sincronización diaria breve (qué hice, qué haré, impedimentos).        |
| **Sprint Review**        | Inspeccionar el incremento y adaptar el backlog con stakeholders.      |
| **Sprint Retrospective** | Mejorar proceso y prácticas del equipo.                                |

## Definición de hecho (DoD)

Criterios que un ítem debe cumplir para considerarse “terminado” (p. ej. código en main, pruebas pasando, documentación actualizada, revisado). Evita que se acumule trabajo “casi terminado” y da transparencia.
