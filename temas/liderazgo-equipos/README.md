# Habilidades para liderar equipos

El liderazgo técnico o de equipo en ingeniería de software abarca responsabilidades sobre entregas, personas y proceso. Las habilidades blandas y de gestión son parte del rol: sin ellas, el equipo y los resultados se resienten aunque el conocimiento técnico sea alto.

## Habilidades necesarias

| Habilidad                    | Descripción breve                                                                 |
| ---------------------------- | --------------------------------------------------------------------------------- |
| Comunicación clara           | Transmitir objetivos, expectativas y criterios de éxito; dar y recibir feedback   |
| Delegación y confianza       | Asignar tareas con criterio, evitar microgestión y permitir que el equipo decida  |
| Escucha activa y empatía     | Entender preocupaciones, limitaciones y motivaciones antes de proponer soluciones |
| Gestión del conflicto        | Detectar tensiones, mediar con imparcialidad y buscar acuerdos basados en hechos  |
| Priorización e incertidumbre | Decidir qué hacer primero con información incompleta y comunicar los trade-offs   |
| Feedback constructivo        | Dar y recibir feedback frecuente, específico y orientado a comportamientos        |
| Organización y visibilidad   | Reuniones útiles, transparencia de estado y bloqueos, métricas entendibles        |

## Retos frecuentes y cómo manejarlos

| Reto                              | Cómo manejarlo                                                                                                                              |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Plazos irrealistas o presión      | Negociar alcance o tiempo con datos (capacidad, dependencias); comunicar riesgos a stakeholders; introducir buffer y priorización explícita |
| Conflictos entre personas o áreas | Escuchar a cada parte por separado; mediar con hechos y objetivos comunes; acuerdos concretos y seguimiento                                 |
| Bajo rendimiento o desmotivación  | 1-1 para entender causas (carga, claridad, habilidades); objetivos claros; apoyo, formación o reasignación según el caso                    |
| Resistencia al cambio             | Explicar el “por qué” y el beneficio; involucrar al equipo en el diseño del cambio; pilotos o rollouts graduales                            |
| “Todo es urgente”                 | Definir criterios de priorización con el negocio; backlog ordenado y visible; decir “no” o “después” con argumentos y alternativas          |

## Prácticas que ayudan

- **Reuniones 1-1:** Espacio periódico con cada miembro para alineación, feedback y bloqueos; no solo status.
- **Retrospectivas:** Revisar qué mejorar en proceso, comunicación y herramientas; actuar sobre acuerdos concretos.
- **Definition of Done:** Criterios compartidos de “terminado” para reducir ambigüedades y rework.
- **Transparencia de métricas:** Velocidad, bloqueos y riesgos visibles para el equipo y para stakeholders, sin usar métricas como arma.
- **Límite entre líder y amigo:** Ser cercano sin comprometer decisiones que afecten al equipo (evaluaciones, asignaciones, correcciones); mantener coherencia y equidad.
